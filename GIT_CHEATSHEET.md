# Шпаргалка по командам Git

## Основные команды

### Создание и клонирование
```bash
git init                    # Создать новый локальный репозиторий
git clone [url]             # Клонировать удаленный репозиторий
```

### Проверка состояния
```bash
git status                  # Показать состояние файлов
git log                     # Показать историю коммитов
git log --oneline           # Краткая история коммитов
```

### Работа с изменениями
```bash
git add [файл]              # Добавить файл в индекс
git add .                   # Добавить все измененные файлы
git add *                   # Добавить все файлы
git commit -m "сообщение"   # Зафиксировать изменения
git commit -am "сообщение"  # Добавить и зафиксировать (только для отслеживаемых файлов)
```

### Работа с ветками
```bash
git branch                  # Показать список веток
git branch [имя]            # Создать новую ветку
git checkout [ветка]        # Переключиться на ветку
git checkout -b [ветка]     # Создать и переключиться на новую ветку
git merge [ветка]           # Слить ветку с текущей
```

### Работа с удаленным репозиторием
```bash
git remote add origin [url] # Добавить удаленный репозиторий
git remote -v               # Показать удаленные репозитории
git push origin [ветка]     # Отправить изменения на сервер
git push -u origin main     # Отправить и установить upstream
git pull origin [ветка]     # Получить изменения с сервера
git fetch                   # Загрузить изменения без слияния
```

### Отмена изменений
```bash
git restore [файл]          # Отменить изменения в файле
git reset HEAD [файл]       # Убрать файл из индекса
git reset --hard [commit]   # Вернуться к определенному коммиту
```

## Жизненный цикл файлов в Git

1. **Untracked** (неотслеживаемый) - новый файл, Git о нем не знает
2. **Staged** (в индексе) - файл добавлен командой `git add`
3. **Committed** (зафиксирован) - файл сохранен в репозитории командой `git commit`
4. **Modified** (изменен) - файл изменен, но не добавлен в индекс

## Полезные советы

- Всегда проверяйте `git status` перед коммитом
- Пишите понятные сообщения коммитов
- Делайте коммиты часто, но логически завершенными
- Перед `git push` выполните `git pull` чтобы избежать конфликтов
